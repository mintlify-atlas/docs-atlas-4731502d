Low-Poly Ocean is a static single-page application (SPA) built with Vite. It can be deployed to any static hosting service, but the repository is pre-configured for **Vercel**.

## Deploying to Vercel

<Steps>
  <Step title="Connect the repository">
    Log in to [vercel.com](https://vercel.com) and click **Add New Project**. Import the repository from GitHub. Vercel detects the Vite framework automatically.
  </Step>

  <Step title="Configure the build settings">
    Vercel's auto-detection sets these values correctly, but verify them under **Build & Output Settings**:

    | Setting | Value |
    |---|---|
    | Build command | `vite build` |
    | Output directory | `dist` |
    | Install command | `npm install` (or your preferred package manager) |
  </Step>

  <Step title="Deploy">
    Click **Deploy**. Vercel runs the build command, uploads the contents of `dist/`, and assigns a `.vercel.app` URL.

    Every push to the `main` branch triggers a new production deployment automatically.
  </Step>
</Steps>

## SPA routing configuration

Because React Router handles all navigation on the client side, the web server must serve `index.html` for every URL — not just `/`. The project includes a `vercel.json` file that configures this behavior:

```json vercel.json
{
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
```

The `source` pattern `/(.*)`  matches every path. The `destination` `/index.html` tells Vercel to serve the app shell for all requests, letting React Router take over and render the correct page in the browser.

<Warning>
  If you deploy to a different host without this rewrite rule (or its equivalent), users who navigate directly to a route like `/detail?id=turtle` or `/stages` will receive a 404 error from the server.
</Warning>

## Manual deployment

To build and deploy the app without Vercel:

<Steps>
  <Step title="Build the app">
    ```bash
    npm run build
    ```

    This runs `vite build` and outputs the production bundle to the `dist/` directory.
  </Step>

  <Step title="Preview the build locally">
    ```bash
    npm run preview
    ```

    Starts a local server that serves the contents of `dist/`. This is the fastest way to verify the production build before uploading.
  </Step>

  <Step title="Upload to a static host">
    Upload the entire contents of `dist/` to your hosting provider. Then configure the equivalent of the Vercel rewrite rule on your host:

    <Tabs>
      <Tab title="Netlify">
        Create a `_redirects` file inside `dist/` (or `public/` before building):

        ```
        /*  /index.html  200
        ```
      </Tab>
      <Tab title="nginx">
        Add a `try_files` directive to your server block:

        ```nginx
        location / {
            try_files $uri $uri/ /index.html;
        }
        ```
      </Tab>
      <Tab title="Apache">
        Create a `.htaccess` file in the document root:

        ```apacheconf
        RewriteEngine On
        RewriteCond %{REQUEST_FILENAME} !-f
        RewriteCond %{REQUEST_FILENAME} !-d
        RewriteRule ^ /index.html [L]
        ```
      </Tab>
    </Tabs>
  </Step>
</Steps>

## What the build produces

```
dist/
├── index.html          # App entry point
├── assets/
│   ├── index-[hash].js   # Bundled JavaScript
│   ├── index-[hash].css  # Bundled CSS
│   └── ...               # Images, audio, and other static assets
```

All filenames in `assets/` include a content hash, so returning visitors' browsers automatically pick up new files after a deployment.

## Environment considerations

The project has no required environment variables. If you add any (see [Configuration](/development/configuration)), set them in the Vercel project's **Environment Variables** settings before deploying — they are injected at build time.

<Note>
  Variables prefixed with `VITE_` are embedded in the compiled JavaScript bundle and are visible to anyone who inspects the source. Never store secrets (API keys, tokens) in `VITE_` variables.
</Note>

## Ignored files

The `.gitignore` ensures that these directories and files are never committed or deployed via source control:

| Ignored path | Reason |
|---|---|
| `node_modules/` | Installed dependencies — regenerated from `package.json` at build time. |
| `dist/` | Build output — generated by `vite build`. |
| `dist-ssr/` | Server-side render output (unused, but excluded by convention). |
| `*.local` | Local environment files (e.g. `.env.local`). |
| `.DS_Store`, `.idea`, `.vscode/*` | Editor and OS metadata. |
